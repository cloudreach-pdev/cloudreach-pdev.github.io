resource organisation models

Conductor has two key concepts which enable organisations to visualise and apply policies to resources flexibly, queries and resource groups. Queries are used to return a collection of resources along a high degree of dimensions, whereas resource groups effectively save these queries into a reusable, logical collection somewhere in your Conductor resource hierarchy. This section outlines some classic approaches to building a resource model to enables your organisation to quickly build a well-defined hierarchy of management and reporting across all integrated services, platforms and resource types.

  • By Cost Center and Application

    Many organisations wish to deliver and manage IT services for their organisation in-line with the company structure and budget lines. This allows policy, cause and effect to be easily traced and all liabilities to have explicit association with purpose to application and data stakeholders. Conductor allows this type of association to be made quickly, as shown below where the organisation is decomposed into several business units and their respective application / service portfolios. Where blurred lines exist, e.g. Engineering and Operations joint manage an application, the same resource group defining that application can be used by both, and Conductors policy resolution engine will automatically mediate any policy conflicts.

  • By Compliance or Governance State

    Quickly establishing views on which resources are in a compliant state is immensely useful for governance and compliance-centric stakeholders. Conductor goes far beyond the ability to simply group resources by tags and allows users to specify sophisticated conditions for defining a resource group such as:

    "All hosted instances / VM's which contain data of classification 'confidential' AND has attached a security group / firewall with port tcp-22 accessible by the world".

    Using these type of queries and resource groups in conjunction with Conductors heatmap dashboard allows your entire security posture and mitigation strategy to be measured and managed in one perspective.

  • By Confidentiality, Integrity and Availability

    The above example models are not mutually exclusive and can be extended as you wish. A typical dimension of management for cloud hosted applications are policies tied to stances around how confidential data is on the systems, required up time or availability. Because Conductor offers solutions which can discover meta-data not just at the cloud provider level, but from build platforms, configuration managers, CMDB's or any API enabled service you wish to integrate: it is possible to use this information to build a comprehensive structure around how resources are grouped and policies applied.
