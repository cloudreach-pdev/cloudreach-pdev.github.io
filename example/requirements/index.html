requirements

Conductor has lean requirements for getting started, namely the following two things are needed:

Platform Setup

Access to an Amazon Web Services (AWS) provider account is required to execute Conductor solutions using AWS Lambda. Conductor simply requires an ARN of an AWS IAM Role meeting or exceeding this policy .

    
    {
       "Version": "2012-10-17",
       "Statement": [
           {
               "Action": [
                   "iam:CreatePolicy",
                   "iam:CreateRole",
                   "iam:AttachRolePolicy"
               ],
               "Effect": "Allow",
               "Resource": "*"
           }
       ]
    }

API Credentials

Services or platforms you wish to integrate Conductor with (for example*) must have a public accessible API, and ability to support authentication via HTTPS or AWS Roles. The Conductor platform is able to manage access or support a self-service model for configuring API credentials.

* This could be a cloud provider such as AWS / Azure, a configuration platform like Ansible / Chef or perhaps an ERP a la Salesforce

Configuring a Resource Provider for Access

For each resource provider used by one or more solutions, configuration to allow access by the solution will be needed, this can be achieved in one of two ways.

  • Platform Managed Access Option A

    Conductor ensures that the least set of permissions are always deployed to service a solution. Where the platform manages access, each resource provider is configured with a master set of credentials (examples below) which are used to generate a secondary set of lesser permissions per solution, as defined by the solution. The benefit is complete automation and maintenance of deployment.

  • User Managed Access Option B

    Conductor allows users to deploy their own permissions per solution, per resource provider. This option is less preferable but well suited to more restricted organisations where all access must undergo review before deployment. In this model, each solution will show what permissions are needed and users expected to configure access out-of-band. The trade-off with this approach is heightened effort and maintenance overall (as permissions will need to be reviewed and updated potentially with every solution update). It is recommended that Platform managed access is used wherever possible to avoid this overhead.
